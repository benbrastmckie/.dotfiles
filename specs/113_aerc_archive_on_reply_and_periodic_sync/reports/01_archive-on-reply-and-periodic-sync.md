## Executive Summary

- **Part A — recommend replacing the drafted `[hooks] mail-sent` + `AERC_SUBJECT` approach with aerc's own native `:send -a flat` archive-on-reply mechanism.** Both are functionally valid for the installed aerc 0.21.0, but the native mechanism is source-confirmed to archive the *exact* message that was replied to (captured by direct reference at `:reply` time), while the hook approach archives whatever is *currently selected* in the active tab at hook-fire time — a real, source-confirmed gap, not a theoretical one, and one that gets *worse*, not better, once Part B's periodic sync is added (new mail arriving mid-compose can reorder/refresh the list the hook will act against).
- **The native mechanism already exists in aerc 0.21.0 and needs only a one-line keybind change**: rebinding `compose::review`'s `y` from `:send<Enter>` to `:send -a flat<Enter>` activates it. It is source-confirmed **inert (never archives)** on `:forward`, fresh `:compose`, and `:recall` — only `:reply`'s (and `:invite`'s) close-handler consumes the `-a` flag — so the rebind is safe to apply unconditionally rather than needing further gating.
- **Both mechanisms must not run simultaneously** — if the hook is kept alongside the native switch, a reply-send would archive the correct message via the native path *and* whatever is separately selected via the hook, potentially archiving an unrelated message. Pick one; this report recommends the native one and recommends deleting the drafted `[hooks] mail-sent` block.
- **Part B — recommend a `systemd.user.services`/`systemd.user.timers` pair running `mail-sync both`, following this repo's own existing `cache-cleanup.nix` pattern exactly**, as the primary "occasional sync regardless of whether aerc is open" mechanism. Optionally *also* wire aerc's `check-mail`/`check-mail-cmd` (`[gmail]`: `check-mail = 10m`, `check-mail-cmd = mail-sync gmail`) for a while-aerc-is-open convenience — `mail-sync`'s flock (task 109) makes concurrent invocations from both triggers fully safe, so "both" is a legitimate, not just tolerated, combination.
- **Two source-confirmed gotchas for the aerc-side `check-mail-cmd` option**: `check-mail-timeout` defaults to only **10s** (`config/accounts.go`), far too short for a network `mbsync` run plus mail-sync's own up-to-300s flock wait — must be raised explicitly (or use `mail-sync gmail --no-wait` in `check-mail-cmd` so a lock-contended call fails fast with a normal, expected error instead of hanging aerc's "Checking for new mail..." UI indicator for minutes). Also, this repo's `u` keybind (`:check-mail<Enter>`) is **already wired but currently errors** ("checkmail: no command specified") because `check-mail-cmd` has never been set — wiring Part B's `check-mail-cmd` fixes this pre-existing broken keybind as a direct side effect, exactly parallel to task 112's `a`/`A` fix.
- **For the systemd-timer option, this repo already has a proven, live, working template** (`cache-cleanup.nix`: oneshot service + `OnCalendar` timer with `Persistent = true`; confirmed active via `systemctl --user list-timers` on this exact host) and a documented cautionary precedent to avoid (`gmail-oauth2.nix`'s *disabled* timer, which permanently "degraded" the systemd --user session because its oneshot job failed on every single run) — the mail-sync case differs materially (expected-success, not permanent-failure), but the failure-visibility trade-off should be called out explicitly rather than silently repeated.

## Findings

### Part A — Archive-on-reply

#### 1. The drafted `[hooks] mail-sent` mechanism is mechanically valid for aerc 0.21.0

- `config/hooks.go` parses `[hooks] mail-sent = ...` into `Hooks.MailSent` (`ini:"mail-sent"`) — the exact key used in the current `aerc.nix` draft. Valid, present, unchanged in this exact build.
- `commands/compose/send.go`'s `sendHelper` calls `hooks.RunHook(&hooks.MailSent{Account, Backend, Header})` **after** the message is sent and copied to Sent, but **before** `composer.Close()`.
- `lib/hooks/mail-sent.go`'s `Env()` sets `AERC_SUBJECT=<outgoing Subject header>` (`m.Header.Get("Subject")`) — this is the header of the message that was just sent, i.e. for a reply it already carries the `"Re: "` prefix aerc's own reply template added (`commands/msg/reply.go` line 145: `subject := "Re: " + trimLocalizedRe(...)`). The draft's `case "$AERC_SUBJECT" in Re:*|RE:*|re:*)` gate is therefore correctly targeting the right variable with the right semantics.
- `aerc :archive flat` invoked from inside a hook script **is** an IPC call into the already-running instance, not a second aerc launch: `main.go` routes any invocation with a non-empty `Command` through `ipc.ConnectAndExec(opts.Command)` unless `--no-ipc`/`general.disable-ipc` is set (neither is set in this config). Confirmed correct as documented in the existing code comment.

#### 2. The "archives the selected message" caveat is real and precisely characterized, not just theoretical

- `aerc :archive flat` (whether typed at the keybind or delivered via IPC from the hook) executes the registered `Archive` command (`commands/msg/archive.go`), whose `Execute()` calls `newHelper().messages()` (`commands/msg/utils.go`).
- `newHelper()` resolves `app.SelectedTabContent()` — **whatever tab is currently active** at the moment the command runs — and `messages()` returns `commands.MarkedOrSelected(...)` against that tab: marked messages if any are marked, otherwise the message under the cursor.
- `sendHelper` calls `app.RemoveTab(composer, false)` **synchronously, immediately** when `:send` is invoked (to "hide the composer from the user" while sending happens in a goroutine) — so by the time the async `mail-sent` hook fires, the active tab has almost certainly already reverted to the message-list tab the user was on before replying. In the ordinary `select → r → reply → send` flow this is indeed the just-replied-to message, matching the existing code comment's claim.
- **However**, nothing pins the cursor to that specific message between `:reply` and the hook firing. If the list reflows during composition (new mail arriving, tags changing sort order, the user switching to another list tab and back), `:archive flat` will act on whatever the cursor/marks point to *then*, not necessarily the original message.
- **This risk compounds with Part B**: enabling periodic sync means new mail can arrive, and the notmuch/maildir list can reflow, *while a reply is being composed*. This is a genuine, non-obvious cross-cutting interaction between the two parts of this task, not present if Part B were absent.
- A secondary, independent flaw of the subject-string gate: it has no way to distinguish "this is actually a reply" from "the user manually typed a `Re:`-prefixed subject into a fresh `:compose`" — the `mail-sent` hook's `Env()` exposes no in-reply-to/reference signal at all, only `AERC_ACCOUNT`, `AERC_ACCOUNT_BACKEND`, `AERC_FROM_*`, `AERC_SUBJECT`, `AERC_TO`, `AERC_CC` — so subject-string matching is the *only* signal available to a `mail-sent` hook, structurally.

#### 3. aerc 0.21.0 has a native, purpose-built "archive the message being replied to" mechanism — more correct than the hook, and already present in this exact build

- `commands/compose/send.go`'s `Send` command already declares `Archive string` with `opt:"-a" action:"ParseArchive" metavar:"flat|year|month" desc:"Archive the message being replied to."` — this is the **documented, canonical** feature for this exact use case (`doc/aerc.1.scd` line 983: `*:send* [*-a* _<scheme>_] ... *-a*: Archive the message being replied to. See *:archive* for schemes.`).
- The flag is not just accepted by `:send` — it is *specifically consumed* by `commands/msg/reply.go`'s `OnClose` callback (lines 196-211):
  ```go
  composer.OnClose(func(c *app.Composer) {
      switch {
      case c.Sent() && c.Archive() != "" && !noStore:
          store.Answered([]models.UID{msg.Uid}, true, nil)
          err := archive([]*models.MessageInfo{msg}, nil, c.Archive())
      ...
  ```
  Critically, `msg` here is the **exact `models.MessageInfo` captured by `widget.SelectedMessage()` at the moment `:reply` was invoked** (line 82, `reply.Execute`) — closed over by reference in the `OnClose` closure — **not** re-resolved from "whatever is selected now." This is immune to any cursor drift, tab switching, or list reflow that happens during composition or after send, including reflow caused by Part B's periodic sync.
  `commands/msg/invite.go` (calendar-invite replies) has the identical pattern.
- **Confirmed inert (never archives) for every non-reply composer path** that could also pass through the same `y`/`:send -a flat` keybind:
  - `:compose` (fresh new message, `commands/account/compose.go`): registers **no** `OnClose` handler at all — `-a flat` is simply unused.
  - `:forward` (`commands/msg/forward.go`, both branches, lines ~188 and ~240): registers `OnClose` but only calls `store.Forwarded(...)`; never reads `c.Archive()`.
  - `:recall` (`commands/msg/recall.go` lines 77-98): registers `OnClose` but only deletes the recalled draft/postponed copy if sent; never reads `c.Archive()`.
- This means the recommended keybind change — `compose::review`'s `y` from `:send<Enter>` to `:send -a flat<Enter>` — is safe to apply **unconditionally** for every send path in this config, with no risk of an unwanted archive on a forward, a fresh compose, or a recalled draft.

#### 4. No collision between `:reply -a` (reply-all) and `:send -a` (archive scheme)

- `reply.All` (`opt:"-a" desc:"Reply to all recipients."`) and `Send.Archive` (`opt:"-a" action:"ParseArchive" ...`) are independent options on two different command structs; aerc's option parser scopes `-a` per-command, so `R` (`:reply -a`, reply-all) and the recommended `y` rebind (`:send -a flat`) coexist with no conflict. A user can press `R` then the rebound `y` and get both reply-all and archive-on-send in the same flow.

#### 5. Trade-off of switching to the native mechanism

- **Gain**: archives the exact replied-to message regardless of any concurrent list reflow, tab switching, or periodic-sync-driven reindex — eliminates the caveat rather than merely accepting it "in the normal flow."
- **Cost**: only takes effect through whatever keybind/alias actually passes `-a flat` to `:send`. A user who bypasses the `y` keybind and types a bare `:send` on the ex command line (an edge case, but a real one) will not auto-archive. The current hook-based draft has the opposite trade-off — it fires regardless of how `:send` was invoked, but can act on the wrong message. This report weighs source-confirmed correctness of *which message gets archived* above coverage of *every possible invocation path*, since a bare typed `:send` bypassing the normal `y` keybind is a rarer, more deliberate action than the reflow/reorder risk the hook carries in the now-more-common case of periodic sync being enabled by Part B.

### Part B — Periodic sync

#### 6. `check-mail`/`check-mail-cmd`/`check-mail-timeout` are valid, documented `accounts.conf` per-account keys for the notmuch backend, in this exact aerc build

- `config/accounts.go`: `CheckMail time.Duration \`ini:"check-mail"\``, `CheckMailCmd string \`ini:"check-mail-cmd"\``, `CheckMailTimeout time.Duration \`ini:"check-mail-timeout" default:"10s"\`` — all per-account (`accounts.conf`), not global `aerc.conf`, consistent with adding them inside the existing `[gmail]`/`[logos]` blocks in `aerc.nix`'s `home.file` text.
- `doc/aerc-accounts.5.scd`: `check-mail = <duration>` — "Maildir/Notmuch folders must use `check-mail-cmd` in conjunction with this option" (this account's `source = notmuch://~/Mail` backend is exactly this case). Default `0` (disabled).
- `doc/aerc-notmuch.5.scd`: `check-mail-cmd = <command>` documented with the example `check-mail-cmd = mbsync -a` — this repo would instead use the flock-serialized, never-`-a` `mail-sync gmail` (or `mail-sync logos`), consistent with every other trigger path in this codebase (`$` keybind, notmuch preNew hook) already routing exclusively through `mail-sync`.
- `worker/notmuch/worker.go`'s `handleCheckMail` (line 768) confirms the *entire* notmuch-backend check-mail behavior is: run `msg.Command` via `sh -c` under a `context.WithTimeout(msg.Timeout)`; on success `w.done(msg)`, on timeout or nonzero exit `w.err(msg, ...)` (surfaced to the UI status line, non-fatal but visible). There is no separate per-folder IMAP-style check for this backend — the shell command *is* the entire mechanism, matching the doc's Maildir/Notmuch carve-out.
- `app/account.go`'s `CheckMail()` guards against re-entrant firing per account (`if acct.checkingMail { return }`) and shows a UI "Checking for new mail..." connection-activity indicator for the duration of the command. This same-account guard does **not** protect against a **separate systemd-timer-triggered** `mail-sync` process running concurrently — that cross-process serialization is exactly what `mail-sync`'s own flock (task 109) already provides, independent of and complementary to this in-process guard.

#### 7. Source-confirmed gotcha: `check-mail-timeout` defaults to 10s — must be raised for this repo's `mail-sync`

- `mail-sync` takes a **blocking flock with up to a 300s wait** before it ever invokes `mbsync`, then runs `mbsync <group>` itself (a network round-trip) plus `notmuch new --no-hooks`. A default 10s `check-mail-timeout` will very likely kill this via `exec.CommandContext`'s context cancellation well before it completes, surfacing a spurious "(gmail) checkmail: timed out" error on the aerc status line even when the sync would otherwise have succeeded.
- **Recommendation**: either (a) set `check-mail-timeout` to something generous (e.g. `60s`-`120s`) in the `[gmail]`/`[logos]` blocks, or (b) point `check-mail-cmd` at `mail-sync gmail --no-wait` so a lock-contended call fails **fast** with a normal, already-anticipated "lock held by another mail-sync run" message (relevant specifically if Part B option (i) and (ii) are both wired — see Finding 9) rather than tying up aerc's "Checking for new mail..." indicator for minutes. Doing both is the most defensible combination: `--no-wait` avoids a long hang, and a moderately raised timeout (e.g. `30s`-`60s`) still comfortably covers a normal, uncontended `mbsync` run over the network.

#### 8. Wiring `check-mail-cmd` also fixes an already-broken, currently-wired keybind (`u`)

- `aerc.nix`'s `messages` keymap already binds `u = ":check-mail<Enter>"` (present before this task, apparently anticipating this exact feature).
- `commands/account/check-mail.go`'s `CheckMail.Execute()` calls `acct.CheckMailReset()` then `acct.CheckMail()`, which posts `types.CheckMail{Command: acct.acct.CheckMailCmd, ...}` to the worker. With `CheckMailCmd` currently unset (empty string), `worker/notmuch/worker.go`'s `handleCheckMail` hits `if msg.Command == "" { w.err(msg, fmt.Errorf("(%s) checkmail: no command specified", ...)) }` — **today, pressing `u` in aerc produces a visible error**, not a no-op. This is directly parallel to task 112's finding that `a`/`A` were silently broken before `maildir-store` was set. Wiring `check-mail-cmd = mail-sync gmail` (or `both`) on `[gmail]` as part of this task makes the pre-existing `u` keybind functional for the first time, as a direct and desirable side effect — worth recording explicitly rather than treating as incidental.

#### 9. Systemd user timer (option ii): this repo has a proven, live template and a documented cautionary precedent

- **Proven template**: `modules/home/services/cache-cleanup.nix` is structurally identical to what Part B option (ii) needs — a `systemd.user.services.<name>` oneshot service (`Type = "oneshot"`, `ExecStart` built via `pkgs.writeShellScript`) plus a matching `systemd.user.timers.<name>` (`OnCalendar`, `Persistent = true`, `RandomizedDelaySec`, `Install.WantedBy = [ "timers.target" ]`, `Unit.Requires = [ "<name>.service" ]`). Confirmed **live and active** on this exact host: `systemctl --user list-timers` shows `cache-cleanup.timer` currently scheduled (next: 2026-07-20, last ran 16h ago successfully).
- `modules/home/services/{screenshot,memory/services}.nix` add a second convention worth reusing: `Unit.After = [ "graphical-session.target" ]` for services that need the graphical/D-Bus session (secret-tool, the credential store `mail-sync`'s underlying `mbsync` calls depend on via `PassCmd`, needs the same).
- `systemctl --user show-environment` on this host confirms `PATH` in the systemd --user instance's own environment already includes `/home/benjamin/.nix-profile/bin` (where `which mail-sync` resolves to `/home/benjamin/.nix-profile/bin/mail-sync`) — so a bare `mail-sync both` in `ExecStart` would already resolve, though this repo's own `cache-cleanup.nix` precedent still exports `PATH` explicitly inside its wrapped script defensively; either approach (explicit `PATH=` export mirroring `cache-cleanup.nix`, or an absolute `${config.home.homeDirectory}/.nix-profile/bin/mail-sync` path mirroring the commented-out `gmail-oauth2.nix` precedent) is consistent with this codebase's existing conventions.
- **Documented cautionary precedent**: `modules/home/services/gmail-oauth2.nix` is a *disabled* (commented-out) systemd user service + timer, left in place specifically as a warning. Its own comment: the refresh job's underlying credential was permanently expired, so it failed on **every single run**, and because the timer fired every 45 minutes, the systemd --user instance was left permanently in a "degraded" state, "visible on each `home-manager switch`." This is a real, previously-experienced operational cost of unconditional systemd-timer failures in *this exact* home-manager setup, not a generic caveat.
  - **Why `mail-sync`'s case differs materially**: unlike the oauth refresh (which was *guaranteed* to fail every time due to a revoked token), `mail-sync both` is expected to succeed under normal conditions (network available, valid Gmail app password / Bridge connection) — a transient failure (laptop offline, Bridge not running) should self-clear on the next successful run rather than persist indefinitely. Still, this should be called out explicitly to the implementer/user as an accepted, monitored trade-off (e.g. periodically check `systemctl --user status mail-sync-timer.service` / `journalctl --user -u mail-sync-timer.service`), not silently assumed away — this repo has already been burned by exactly this failure mode once.
- **Recommended interval**: the task states "occasional/non-immediate sync is acceptable," so something in the 15-30 minute range (e.g. `OnCalendar = "*:0/15"` or `*:0/30`) is a reasonable, easily-adjustable starting point — smaller than `cache-cleanup.timer`'s `weekly` cadence (a very different workload) but far coarser than any near-real-time expectation. `Persistent = true` ensures a missed run (laptop asleep through the interval) fires promptly on next wake, matching `cache-cleanup.timer`'s existing convention.

#### 10. Flock-safety of running both options (i) and (ii) simultaneously

- `mail-sync`'s flock (`modules/home/email/mail-sync.nix`) is acquired **before any `mbsync` invocation, per-process, regardless of trigger** — this is precisely the invariant task 109 built it for (the notmuch preNew hook and aerc's `$` keybind already share this same lock safely today). A systemd-timer-triggered `mail-sync both` and an aerc-`check-mail`-triggered `mail-sync gmail` contending for the same lock will simply serialize: one waits (up to 300s by default) or fails fast if `--no-wait` is used.
- The only place this interacts with correctness (not just performance) is the `check-mail-timeout` sizing discussed in Finding 7 — a contended wait inside aerc's `check-mail-cmd` must either be given enough timeout headroom to wait out the other run, or be told to fail fast via `--no-wait` and rely on the next scheduled `check-mail`/timer tick to retry. Recommend `--no-wait` for the aerc-side invocation specifically *if* both options are wired together, since the systemd timer will pick up the sync shortly regardless and there is no need to make aerc's UI-visible "Checking for new mail..." indicator block for up to 5 minutes.
- Both options are legitimately combinable, not just "tolerated" — recommend wiring both: the systemd timer as the unconditional, always-on guarantee (satisfies "Gmail-in-the-browser stays in sync occasionally" even when aerc is closed), and the aerc `check-mail`/`check-mail-cmd` pair as a lightweight while-open convenience that also happens to fix the pre-existing broken `u` keybind (Finding 8).

#### 11. `notmuch new`'s preNew hook already calls `mail-sync both`, but only on-demand

- `modules/home/email/notmuch.nix`'s `preNew = "mail-sync both || true"` (task 109) means *any* hook-ful `notmuch new` invocation already triggers a full both-account sync today — but nothing currently invokes plain `notmuch new` (with hooks) on any schedule; every existing automated path uses `notmuch new --no-hooks` specifically to *avoid* re-triggering this hook (mail-sync's own internal reindex, `email-reindex`). This confirms there genuinely is no existing periodic trigger today, and that Part B's systemd timer calling `mail-sync both` directly (which itself ends with `notmuch new --no-hooks`) is the correct, hook-bypassing way to add one — not, e.g., a timer that calls bare `notmuch new` (which would be redundant with, and race, the preNew hook's own `mail-sync both` call).

## Decisions

- **Part A**: recommend switching to aerc's native archive-on-reply mechanism — rebind `extraBinds.compose::review.y` from `":send<Enter>"` to `":send -a flat<Enter>"`, and **remove** the drafted `[hooks] mail-sent` block rather than keeping both (running both risks a double-archive: the correct message via the native path, plus whatever is separately selected via the hook). This is grounded in `commands/msg/reply.go`'s `OnClose` closing over the exact replied-to `models.MessageInfo` by reference, which is immune to the list-reflow risk the hook approach carries and which gets worse once Part B is added.
- **Part B**: recommend wiring both mechanisms:
  - A `systemd.user.services`/`systemd.user.timers` pair (following `cache-cleanup.nix`'s exact shape) running `mail-sync both` on an interval in the 15-30 minute range, `Persistent = true`, `After = [ "graphical-session.target" ]` — this is the unconditional guarantee that satisfies "stays in sync occasionally" regardless of whether aerc is open.
  - `[gmail]` (and optionally `[logos]`) `check-mail = 10m` + `check-mail-cmd = mail-sync gmail` (`mail-sync logos` for `[logos]`) as a while-aerc-is-open convenience, with `check-mail-timeout` explicitly raised well above the 10s default (or `--no-wait` used in the command itself) to avoid spurious timeout errors against `mail-sync`'s up-to-300s flock wait plus a real network `mbsync` round-trip.
- Both are safe to run together because of `mail-sync`'s existing flock (task 109); no new locking mechanism is needed for this task.

1. **(Part A, primary)** Change `modules/home/email/aerc.nix`'s `extraBinds.compose::review.y` to `":send -a flat<Enter>"`; delete the `[hooks].mail-sent` entry (and the `hooks = { ... }` block entirely if nothing else populates it). Document the "why" inline (native mechanism archives the exact replied-to message by reference; the hook-based alternative was considered and rejected for the list-reflow risk detailed in Finding 2, which worsens once Part B is added).

Suggested starting interval: `OnCalendar = "*:0/15"` (every 15 minutes) or `*:0/30`; adjust to taste since "occasional" is explicitly acceptable.
3. **(Part B, secondary/optional)** Add `check-mail = 10m`, `check-mail-cmd = mail-sync gmail --no-wait`, `check-mail-timeout = 30s` (or similar) to `[gmail]` in `accounts.conf`; optionally the same for `[logos]` with `mail-sync logos --no-wait`. 5. Note for the implementer: no live-mail verification is performed by this research; the implementer should build-verify only (`home-manager build`), and defer the reply→archive→sync→Gmail-web-UI end-to-end check to the user, per this task's own non-goal.

## Risks & Mitigations

- **Risk**: switching Part A to the native mechanism changes user-visible behavior for anyone who currently types a bare `:send` on the ex command line instead of using `y` — that path will no longer auto-archive. **Mitigation**: this is a deliberate, documented trade-off (Finding 5); the normal keybind-driven flow is unaffected, and the tradeoff is in favor of correctness (right message archived) over invocation-path coverage.
- **Risk**: `check-mail-timeout`'s 10s default silently truncates `mail-sync`'s intended flock-wait/network-round-trip behavior, producing confusing spurious timeout errors on the aerc status line. **Mitigation**: explicitly set a generous `check-mail-timeout` and/or use `mail-sync gmail --no-wait` in `check-mail-cmd`, per Finding 7.
- **Risk**: a systemd-timer-triggered `mail-sync both` failing (offline laptop, Bridge down) could leave the systemd --user instance in a "degraded" state, as previously experienced and explicitly documented in this repo's disabled `gmail-oauth2.nix` timer. **Mitigation**: this repo's precedent is a *permanent*, guaranteed-every-run failure (expired token) — `mail-sync`'s expected-success profile is materially different (transient, self-clearing on next success) — but the implementer/user should still know to check `systemctl --user status`/`journalctl --user` occasionally rather than assume silent success, consistent with this project's "never silent" philosophy.
- **Risk**: running both Part B mechanisms together could make aerc's `check-mail` block for up to 300s if it loses the flock race against a timer-triggered `mail-sync both`. **Mitigation**: use `--no-wait` on the aerc-side invocation specifically when both mechanisms are wired (Finding 10).
- **Risk (identified, not fully resolved by this research)**: whether aerc's in-memory message-list selection/cursor is stable across a notmuch reindex triggered by a *concurrent* `mail-sync` while a reply composer is open is not verified at the source level in this pass (would require live TUI testing, out of scope for research). This is exactly the scenario Finding 2 flags as the reason to prefer the native archive-on-reply mechanism over the hook-based one — the native mechanism sidesteps needing an answer to this question entirely, since it never re-resolves "current selection" after `:reply` is invoked.
