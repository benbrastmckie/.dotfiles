## Overview

Implement Recommendation A from the research report: add `<Tab>` / `<S-Tab>`
tab-switching to the aerc `[messages]` and `[view]` contexts in
`modules/home/email/aerc.nix`, so the Neovim reflex "`<Tab>`/`<S-Tab>` = switch
buffers/tabs" transfers to aerc account tabs. The existing `<C-n>` / `<C-p>`
global binds are kept verbatim as guaranteed-reachable fallback aliases (they
also cover compose contexts, where `<Tab>` is deliberately not stolen). The
change is confined to `.dotfiles` (a single Home Manager module), carries zero
cross-repo risk, and must not regress any of the email-safety binds. Definition
of done: `home-manager build` evaluates cleanly, a `home-manager switch`
regenerates the `binds.conf` symlink, `<Tab>`/`<S-Tab>` switch tabs in the
message list and message view, and every safety-critical bind (`d`/`D`/`a`/`A`,
the `Proposed-*` folder overrides, the scoped `$` sync) behaves exactly as
before.

### Research Integration

Key findings integrated from `reports/01_aerc-keymap-alignment.md`:
- Section 4.1 key-reachability table: `<Tab>`/`<S-Tab>` pass straight through
  the Neovim terminal layer to aerc (buffer-nav maps are normal-mode only), so
  Ask 1 is viable purely in aerc.nix. This is why the change is
  `.dotfiles`-only.
- Section 6.1 Recommendation A: bind `<Tab>`/`<S-Tab>` in the read contexts
  (`[messages]`, `[view]`), NOT in `global`/`compose` (aerc `global` binds are
  inherited into `[compose]`, where `<Tab>` may be needed in text fields / the
  review screen). Keep `<C-n>`/`<C-p>` as aliases.
- Section 7 safety invariants: `d`/`D`/`a`/`A` are human-only mutation paths;
  the `Proposed-Delete/Archive/Unsure` folder overrides route through the
  `email-classify --append-approved` wrapper flow; `$` is deliberately
  group-scoped and hook-bypassing. None of these are touched by adding Tab
  binds.
- Section 8 verification plan and the WezTerm backtab (`CSI Z`)
  distinguishability check are adopted verbatim in Phase 2.

## Goals & Non-Goals

**Goals**:
- Add `<Tab>` = `:next-tab<Enter>` and `<S-Tab>` = `:prev-tab<Enter>` to the
  aerc `[messages]` context in `modules/home/email/aerc.nix`.
- Add the same two binds to the aerc `[view]` context.
- Preserve the existing `[global]` `<C-n>`/`<C-p>` binds as fallback aliases
  (no change to them).
- Verify config evaluates and deploys, tabs switch, and no safety bind regresses.

**Non-Goals**:
- Recommendation B (unblocking `<C-h/j/k/l>` and `<Esc>` inside aerc). That is a
  cross-repo change to `~/.config/nvim/lua/neotex/config/keymaps.lua` and is
  explicitly out of scope for this `.dotfiles` task; it should be filed as a
  separate dependent Neovim task if the `<C-hjkl>` payoff is wanted.
- Adding `<Tab>`/`<S-Tab>` to `[global]`, `[compose]`, `compose::editor`, or
  `terminal` contexts (would risk stealing `<Tab>` from compose text fields /
  the review screen).
- Adopting an `s` = sync alias (research Open Question 3). Deferred to keep the
  change minimal and focused; can be a trivial follow-up reusing the exact
  scoped `$` exec form.
- Retiring `<C-n>`/`<C-p>` (research Open Question 5 recommends keeping them
  permanently as zero-cost fallbacks).
- Any change to `binds.conf` directly (it is a read-only nix-store symlink,
  regenerated by `home-manager switch`).

## Risks & Mitigations

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Terminal (WezTerm) does not emit a distinct backtab, so aerc never sees `<S-Tab>` | L | M | `<C-p>` alias remains a working fallback; Phase 2 explicitly tests `<S-Tab>` and documents the terminal keyboard-protocol caveat if it fails. `<Tab>` (forward) is unaffected. |
| Accidentally regressing a safety-critical bind (`d`/`D`/`a`/`A`, `Proposed-*`, `$`) | H | L | Change is purely additive (two new keys per context); Phase 2 has an explicit regression checklist for every safety bind. No existing line is edited. |
| Adding `<Tab>` where it collides with compose/text entry | M | L | Binds scoped strictly to `[messages]` and `[view]`; `[compose]`/`compose::editor`/`terminal`/`global` left untouched (see Non-Goals). |
| `home-manager switch` fails to regenerate the symlink | M | L | Phase 2 runs `home-manager build` first (pure eval, no activation) before `switch`; on failure the old symlink stays intact and the change is reverted per Rollback. |

## Rollback/Contingency

- The change is a small additive diff to a single file. To revert: `git checkout
  modules/home/email/aerc.nix` (before commit) or `git revert` the commit (after),
  then `home-manager switch` to regenerate the previous `binds.conf`.
- If `home-manager build` fails at Phase 2, the current symlink is untouched
  (build does not activate); fix the syntax in aerc.nix and rebuild, or revert.
- If `<S-Tab>` proves undeliverable by the terminal, no rollback is needed:
  `<C-p>` remains the working prev-tab fallback; document the caveat and
  optionally file a terminal keyboard-protocol follow-up.
