# Research Report: Aligning aerc Keybindings with Neovim / Himalaya Conventions

**Task**: 105 — aerc_keybindings_nvim_himalaya_alignment
**Type**: nix (Home Manager module edit)
**Date**: 2026-07-08
**Status**: Research complete

---

## 1. Goal

Bring the aerc terminal email client's keybindings into alignment with the
navigation conventions the user already uses in Neovim and the Himalaya
Neovim plugin, so muscle memory transfers across all three. The two concrete
asks that seeded this task:

1. **Tab switching** — use `<Tab>` / `<S-Tab>` (the Neovim "next/prev buffer"
   keys) to switch aerc account tabs, instead of the current `<C-n>` / `<C-p>`.
2. **Pane navigation** — the user uses `<C-h/j/k/l>` to move between Neovim
   windows and would like aerc navigation to feel the same.

Both asks turn out to have a shared dependency: **aerc runs inside a Neovim
terminal**, so Neovim's own terminal-mode mappings sit *between* the keyboard
and aerc and can swallow keys before aerc ever sees them. Section 4 is the
crux of this report.

## 2. The three keymap systems being reconciled

### 2.1 Neovim core navigation (the anchor conventions)
Source: `~/.config/nvim/lua/neotex/config/keymaps.lua`

| Keys | Action | Mode |
|------|--------|------|
| `<C-h>` / `<C-j>` / `<C-k>` / `<C-l>` | Move between windows (`wincmd h/j/k/l`) | normal **and** terminal |
| `<Tab>` / `<S-Tab>` | Next / previous buffer (mtime-sorted) | normal only |
| `<C-d>` / `<C-u>` | Half-page down / up | normal |
| `<C-f>` / `<C-b>` | Full page down / up | normal |
| `<Esc>` (in terminals) | Exit terminal mode (`<C-\><C-n>`) | terminal |

Mental model: **`<C-hjkl>` = panes, `<Tab>`/`<S-Tab>` = buffers/tabs.**

### 2.2 Himalaya Neovim plugin sidebar
Source: `~/.config/nvim/lua/neotex/plugins/tools/himalaya/config/ui.lua`
and `.../ui/folder_help.lua`

| Key | Action |
|-----|--------|
| `j` / `k` | Next / prev email |
| `A` | Switch account (picker via `vim.ui.select`) |
| `s` / `S` | Sync inbox / full sync |
| `i` | Sync info |
| `u` | Toggle read/unread |
| `*` | Toggle star/flagged |
| `l` / `L` | Apply label / label management |
| `<Tab>` / `<S-Tab>` | Toggle thread / toggle all threads |
| `<CR>` | Preview (3-state) |
| `r` / `R` / `f` | Reply / reply-all / forward |
| `d` / `a` / `c` | Delete / archive / change folder |
| `?` | Context-aware help |
| `q` | Quit sidebar |

Note the one genuine tension: **Himalaya overloads `<Tab>` for thread
expand/collapse**, whereas the user's Neovim anchor uses `<Tab>` for
buffer/tab switching. See §5 for why we align aerc to the Neovim anchor,
not to Himalaya's thread overload.

### 2.3 aerc (current state)
Source of truth: `~/.dotfiles/modules/home/email/aerc.nix` (`extraBinds`).
The runtime file `~/.config/aerc/binds.conf` is a **read-only nix-store
symlink** — never edit it directly; edit the `.nix` and `home-manager switch`.

Current `[global]`: `<C-n>` = next-tab, `<C-p>` = prev-tab, `<C-t>` = term,
`?` = help.
`[messages]` already mirrors vim well: `j/k`, `g/G`, `<C-d>/<C-u>/<C-f>/<C-b>`,
`r/R/f`, `c`, `/`, `n/N`, `<Space>`. Folder movement is `J` / `K`
(`:next-folder` / `:prev-folder`).

## 4. CRITICAL FINDING — Neovim's terminal layer intercepts keys before aerc

aerc is not a bare terminal app here; it runs inside Neovim. Neovim installs
buffer-local **terminal-mode** mappings on *every* terminal via a `TermOpen`
autocmd with `pattern = { "term://*" }`
(`~/.config/nvim/lua/neotex/config/autocmds.lua:25`), which calls
`set_terminal_keymaps()` (`.../keymaps.lua:116`). aerc is neither the Claude
nor OpenCode special case, so it hits the `else` branch
(`keymaps.lua:147-150`):

```
t  <C-h>  ->  <Cmd>wincmd h<CR>
t  <C-j>  ->  <Cmd>wincmd j<CR>
t  <C-k>  ->  <Cmd>wincmd k<CR>
t  <C-l>  ->  <Cmd>wincmd l<CR>
```

and (`keymaps.lua:133`):

```
t  <Esc>  ->  <C-\><C-n>   (exit terminal mode)
```

### 4.1 Key reachability table (does aerc receive the key?)

| Key(s) | Intercepted by Neovim terminal mode? | Reaches aerc? |
|--------|--------------------------------------|---------------|
| `<C-h>` `<C-j>` `<C-k>` `<C-l>` | **Yes** → `wincmd` | **No** |
| `<Esc>` | **Yes** → exits terminal mode | **No** (aerc never sees Esc) |
| `<Tab>` `<S-Tab>` | No (buffer-nav maps are *normal* mode only) | **Yes** |
| `<C-n>` `<C-p>` | No | **Yes** (this is why current tab-switch works) |
| `<C-d>` `<C-u>` `<C-f>` `<C-b>` | No | **Yes** |
| `<C-t>`, letters, `/`, `<Space>` | No | **Yes** |

### 4.2 Consequences for the two asks

- **Ask 1 (`<Tab>`/`<S-Tab>` for tabs): VIABLE.** These keys pass straight
  through to aerc. We simply bind them in aerc to `:next-tab` / `:prev-tab`.
- **Ask 2 (`<C-hjkl>` inside aerc): BLOCKED at the Neovim layer.** aerc can
  never receive `<C-hjkl>` unless we special-case the aerc terminal in
  Neovim's `set_terminal_keymaps()` (a **cross-repo** change in
  `~/.config/nvim`, not `.dotfiles`). Moreover, aerc has **no focusable-pane
  model** — its "panes" (folder sidebar / message list / message view) are not
  independently focusable; you move between folders with commands
  (`:next-folder`/`:prev-folder`), not window focus. And because the aerc
  window is a *fullscreen float* with no sibling Neovim windows, the current
  `wincmd h/j/k/l` do nothing useful there anyway.
- **Side finding (`<Esc>`): usability wrinkle.** aerc uses `<Esc>` to cancel
  prompts/selections (e.g. the `Archive ALL marked?` confirm, `:search`,
  active selections). Today `<Esc>` exits Neovim terminal mode instead of
  reaching aerc, so those prompts can't be dismissed with Esc from inside the
  float. Worth addressing in the cross-repo option (§6.2).

## 5. Design principles adopted

1. **Anchor to the Neovim core model, not Himalaya's overloads.** Neovim is
   the dominant daily driver; its `<C-hjkl>=panes`, `<Tab>=tabs/buffers` model
   is the user's reflex. aerc *tabs* are conceptually Neovim *buffers/tabs*, so
   `<Tab>`/`<S-Tab>` → tab switching is the most transferable mapping.
   Himalaya's `<Tab>`=thread-toggle is a local exception we do not propagate.
2. **Never regress safety-critical binds** (see §7). Alignment is cosmetic;
   the email-safety design is not negotiable.
3. **Prefer single-repo, low-risk changes** (aerc.nix only) and quarantine any
   cross-repo Neovim change as an explicit, optional follow-up.
4. **Additive over destructive**: keep `<C-n>`/`<C-p>` as working aliases when
   adding `<Tab>`/`<S-Tab>`, so nothing breaks mid-transition and there is a
   guaranteed-reachable fallback if a terminal doesn't emit a clean `<S-Tab>`.

## 6. Proposed scheme

### 6.1 Recommendation A — aerc.nix only (LOW RISK, recommended baseline)

Edit `~/.dotfiles/modules/home/email/aerc.nix` `extraBinds`:

**Add `<Tab>`/`<S-Tab>` tab switching to the read contexts** (message list and
message view), keeping `<C-n>`/`<C-p>` as global aliases:

```nix
# [global] — keep existing as fallback aliases
"<C-p>" = ":prev-tab<Enter>";
"<C-n>" = ":next-tab<Enter>";

# [messages] — add:
"<Tab>"   = ":next-tab<Enter>";
"<S-Tab>" = ":prev-tab<Enter>";

# [view] — add the same two:
"<Tab>"   = ":next-tab<Enter>";
"<S-Tab>" = ":prev-tab<Enter>";
```

**Deliberately NOT adding `<Tab>` to `global` or the compose contexts.**
Rationale: aerc `global` binds are inherited into `[compose]` (no
`$noinherit`), where `<Tab>` may be needed inside text fields / the review
screen. Scoping `<Tab>`/`<S-Tab>` to the read contexts avoids stealing Tab
from composition. `<C-n>`/`<C-p>` remain the "works everywhere including
compose" path.

**Optional consistency touch-ups (aerc.nix, still low risk):**
- Add `s` = the existing safe sync exec as an alias of `$`, to match
  Himalaya's `s`=sync reflex — **only** re-using the already-scoped
  `mbsync gmail && notmuch new --no-hooks` form (never `mbsync -a`; see §7).
  Trade-off: `s` is currently unbound in `[messages]`, so this is free.
- Leave folder nav on `J`/`K` (already shift-vim style; consistent enough and
  collision-free).

Net effect of Recommendation A: Ask 1 fully satisfied; Ask 2 acknowledged as
not achievable without §6.2; zero cross-repo risk; zero safety regression.

### 6.2 Recommendation B — cross-repo, unlock `<C-hjkl>` + fix `<Esc>` (OPTIONAL follow-up)

This is a **separate task in `~/.config/nvim`** (the neovim repo), because it
edits `set_terminal_keymaps()`, not aerc.nix. Only pursue if Ask 2 is truly
wanted.

1. In `~/.config/nvim/lua/neotex/config/keymaps.lua`, add an `is_aerc`
   detection (mirroring `is_claude` / `is_opencode`, e.g. match the terminal
   bufname/cmd for `aerc`) and, for aerc terminals:
   - **Skip** the `<C-h/j/k/l>` → `wincmd` remap so they reach aerc.
   - **Skip** the `<Esc>` → exit-terminal-mode remap so aerc prompts cancel
     with Esc (like the existing `is_claude` exemption).
   (Safe because the aerc float has no sibling windows to `wincmd` into.)
2. Then in aerc.nix, bind inside aerc, e.g. a spatial scheme:
   - `<C-h>` = `:prev-folder<Enter>`, `<C-l>` = `:next-folder<Enter>`
     (folders live in the left sidebar → horizontal keys), **or** keep folders
     on `J`/`K` and use `<C-j>`/`<C-k>` = `:next`/`:prev` message.
   Exact assignment is a planning decision; both are internally consistent.

Cost: touches a second repo, changes shared terminal behavior (must confirm no
regression for other terminals — the change is gated on `is_aerc`, so risk is
contained). Benefit: true `<C-hjkl>` + working `<Esc>` inside aerc.

**Recommendation: ship A now; file B as a dependent follow-up task and decide
during planning whether the C-hjkl payoff justifies the cross-repo edit.**

## 7. Safety-critical binds that MUST be preserved (do not regress)

The aerc.nix comments encode deliberate email-safety decisions (`.dotfiles`
task 72 Phase 9, referenced in `handoffs/mail-29-runbook.md`). Any keymap
edit must leave these behaviors intact:

- **`d` / `D` / `a` / `A`** in `[messages]` are **human-only** mutation paths,
  intentionally outside the agent guardrail (the `mail-guard.sh` PreToolUse
  hook can only gate Claude's own Bash calls, not aerc's Go worker).
  `D` (bare delete) and `A` (bulk archive) are hardened with `:prompt`
  confirms; `d` already prompts. **Do not** repurpose these letters for
  navigation. In particular, aerc `A` = "archive ALL marked" — this is why `A`
  cannot be reused for account switching (which is exactly why `<Tab>`/`<S-Tab>`
  is the right home for tab/account switching).
- **`Proposed-Delete` / `Proposed-Archive` / `Proposed-Unsure` folder
  overrides**: `d`/`a`/`k` in those views retag `+confirmed-*` / `+proposed-keep`
  and `:exec email-classify --append-approved {{.MessageId}}`. They route
  through the wrapper/manifest approval flow and must never be replaced by
  aerc-native `:delete-message` / `:archive`.
- **`$` sync scoping**: `:exec mbsync gmail && notmuch new --no-hooks` is
  deliberately group-scoped and hook-bypassing (never `mbsync -a`, never a
  hook-triggering `notmuch new`) to avoid the Logos/Bridge freeze-blast-radius.
  Any `s` alias (§6.1) must reuse this exact form.

None of the proposed changes in §6.1 touch these; they only add `<Tab>`,
`<S-Tab>`, and optionally an `s` alias to the same safe exec.
