## Executive Summary
- All three symptoms are confirmed by direct file inspection: `config/config.kdl` autostart section (lines 249-261) has no waybar spawn and no polkit agent spawn, and the wallpaper line (250) uses `~/.wallpapers/current`, which niri's non-shell `spawn`/`spawn-at-startup` never expands.
- **Waybar**: fix is `spawn-at-startup "waybar"` in `config/config.kdl`. Confirmed via home-manager's `programs.waybar` module source: `systemd.enable` defaults to off (nothing currently starts it), and if enabled its unit's default target list is `[ config.wayland.systemd.target ]`, which itself defaults to the generic `graphical-session.target` — and niri's own `niri.service` unit `BindsTo=graphical-session.target`/`Before=graphical-session.target`, confirming that target is compositor-agnostic and shared across whichever session (GNOME or niri) activates it, so the double-bar risk is real and structurally grounded, not speculative.
- **Polkit agent**: fix is to add a package that exposes `polkit-gnome-authentication-agent-1` on a stable, non-store path and `spawn-at-startup` it. Live-built `pkgs.polkit_gnome` (0.105) confirms the binary lives at `<pkg>/libexec/polkit-gnome-authentication-agent-1` (matches the task's assumption) — but `libexec` is **not** one of the directories NixOS's system profile (`/run/current-system/sw`) links (only `bin`, `sbin`, `lib`, `etc`, `share` are linked, confirmed by listing `/run/current-system/sw`), so a bare `pkgs.polkit_gnome` install gives no stable invocable path. The repo's own `writeShellScriptBin` wrapper pattern (already used twice in this same file, for `zathura`/`sioyek`) is the correct fix: wrap the libexec binary in a `pkgs.writeShellScriptBin "polkit-gnome-authentication-agent-1"` derivation added to `environment.systemPackages`, giving a stable `bin/`-linked path `/run/current-system/sw/bin/polkit-gnome-authentication-agent-1`, spawned exactly like the existing `swaybg` line.
- Also confirmed: the shipped `polkit-gnome-authentication-agent-1.desktop` XDG autostart entry has `OnlyShowIn=GNOME;XFCE;Unity;` and `AutostartCondition=GNOME3 unless-session gnome`, so niri's XDG-autostart-on-systemd-session feature (`niri.service` `Wants=xdg-desktop-autostart.target`) will **not** auto-launch it under niri (ruling out a "no-code" fix), and it will **not** double-register under a real GNOME session either (ruling out a conflict risk from simply having the package installed system-wide).
- **Wallpaper**: niri's bundled wiki confirms `spawn-at-startup`/`spawn` never invoke a shell (no `~` expansion) and a separate `spawn-sh-at-startup` exists specifically for shell semantics (since niri 25.08, present in the installed 26.04). Since `~/.wallpapers/current` is never created by any module (only `/etc/wallpapers/riverside.jpg` exists, via `modules/system/desktop.nix:33`), switching to the absolute path is strictly correct — a `spawn-sh-at-startup` wrapper would only fix the `~` expansion, not the missing target file, so it does not solve the actual bug on its own.

## Context & Scope

Scope is strictly the three fixes named in the task, all confined to the niri-only surface (`config/config.kdl`'s autostart section and, for polkit, `modules/system/packages.nix`). GNOME's own session config (`modules/system/desktop.nix`) and GNOME-only services are out of scope and were only inspected to confirm they are not accidentally affected by the niri-side changes.

`config/config.kdl` is wired into the live config via `modules/home/core/shell.nix:25`:
```
".config/niri/config.kdl".source = ../../../config/config.kdl;
```
This is a **plain static file symlink** (`home.file.source`, not `xdg.configFile.text` with Nix-side templating). Any recommendation that needs a Nix-computed value inside `config.kdl` (i.e., the polkit agent path) must therefore go through a package that resolves to a **stable, predictable path** rather than needing string substitution inside the KDL file itself — this is the deciding factor in the polkit recommendation below.

## Findings

### 1. Waybar not started (config.kdl / waybar.nix)

- `modules/home/desktop/waybar.nix:4-5` sets `programs.waybar.enable = true;` with a full `settings.mainBar` block (workspaces/window/clock/tray/battery/network/pulseaudio/bluetooth/idle_inhibitor modules) but never sets `programs.waybar.systemd.enable`.
- Read the live home-manager module source (`modules/programs/waybar.nix` in the pinned `home-manager` flake input, rev `7bfff44b`): `systemd.enable` is an `mkEnableOption` (defaults `false`). When `false`, the module's `config` block only writes `xdg.configFile."waybar/config"` / `"waybar/style.css"` and adds `cfg.package` to `home.packages` — **it never starts the process**. This exactly matches the task's diagnosis: nothing spawns waybar, and it is not D-Bus-activatable (no D-Bus service file is installed for it), so the reserved 32px strut at `config/config.kdl:57` (`layout.struts.top 32`) produces a persistent empty gap.
- `config/config.kdl`'s autostart section (lines 243-261) has no `waybar` line at all, and no other module (kanshi, mako, swaylock) launches it either.
- **If `programs.waybar.systemd.enable = true` were used instead** (the alternative the task pre-emptively rejects): the module's `systemd.targets` option (`modules/programs/waybar.nix:238-250`) defaults to `[ config.wayland.systemd.target ]`, and `wayland.systemd.target` (`modules/wayland.nix:8-21`) itself defaults to the literal string `"graphical-session.target"`. Cross-checked against the **installed niri package's own shipped unit** (`/nix/store/.../niri-26.04/share/systemd/user/niri.service`):
  ```
  [Unit]
  BindsTo=graphical-session.target
  Before=graphical-session.target
  ```
  This proves `graphical-session.target` is the generic, compositor-agnostic systemd user target — it is not niri-specific. GNOME's own Wayland session integration binds/starts the same target name. Since there is one shared `systemd --user` manager per Linux user (not one per login session), any unit `WantedBy=graphical-session.target` that starts while the target is active in a niri login will remain running (or be restarted) if the user later logs into GNOME without the target being cleanly torn down between sessions, producing the "stray second bar in GNOME" the task describes. **This is a structurally real risk, not a guess** — it is exactly the reason the repo's own `kanshi.nix` (`modules/home/desktop/kanshi.nix:6-7`) already overrides `services.kanshi.systemdTarget = "niri.service"` (a genuinely niri-scoped target, confirmed to exist as a real installed unit) instead of accepting the home-manager default.
- **Recommendation (primary, matches task directive)**: add to the "AUTOSTART PROGRAMS" section of `config/config.kdl`:
  ```
  spawn-at-startup "waybar"
  ```
  Bare command name (no path) is safe: `waybar` is only installed via `home.packages` (through `programs.waybar.enable`), and `lib/mkHost.nix:42-43` sets `home-manager.useUserPackages = true;`, which registers the per-user profile (`/etc/profiles/per-user/benjamin/bin`) into the system-wide `PATH` via the NixOS home-manager module — the same mechanism that already makes bare `spawn-at-startup "kitty"` (line 253) work today (though kitty also happens to be a system package). This also matches upstream niri's own shipped `default-config.kdl:271`, which uses the identical bare `spawn-at-startup "waybar"` line.
  - Placement: this line reserves and fills exactly the struts space already declared at `config.kdl:57` (`top 32`), matching waybar's configured `height = 32` (`waybar.nix:10`) — no strut adjustment needed.
  - The stale comment at `config.kdl:245-247` ("systemd handles service activation via D-Bus (no manual spawns needed)... See: specs/plans/010_niri_gnome_portal_integration.md") should be corrected/removed as part of implementation — it is inaccurate for waybar (no D-Bus activation exists for it) and cites a plan not driving this config; task 78 (`niri_documentation_rewrite`) also flags stale cross-references of this kind, so a one-line comment fix here is low-risk and directly relevant.
- **Alternative considered and correctly rejected by the task**: `programs.waybar.systemd.enable = true;` bound to the default `graphical-session.target` — confirmed above to be genuinely shared with GNOME. A scoped variant (`programs.waybar.systemd.targets = [ "niri.service" ];`, mirroring kanshi's own `systemdTarget = "niri.service"` pattern) *would* work and is more idiomatic with this repo's existing HM-systemd convention, and gains `systemctl --user status/restart waybar.service` observability (per niri's own `Example-systemd-Setup.md`) — but it is a larger structural change (adds a new systemd unit, changes reload semantics for `programs.waybar.style`) than the one-line `spawn-at-startup` fix the task specifies. Flagging as a future alternative, not a countermanding recommendation.

### 2. No polkit agent (packages.nix / config.kdl)

- `modules/system/packages.nix:32` has `polkit_gnome` commented out inside a "For use with Niri without Gnome utilities" block (lines 23-35) that also lists mako/grim/slurp/swaylock/waybar/swayidle/kanshi — all of which are *actually* provided elsewhere via home-manager modules already (`mako.nix`, `kanshi.nix`, `swaylock.nix`, `waybar.nix`, and grim/slurp/satty via `modules/home/packages/media-dictation.nix`). `polkit_gnome` is the one item in that block with no live equivalent anywhere in the repo (`grep -rn polkit` across the repo turns up only `security.polkit.enable = true;` at `modules/system/desktop.nix:57`, which enables the polkit **daemon/authority**, not an authentication **agent** — these are distinct polkit components).
- Confirmed via live `nix build nixpkgs#polkit_gnome`: the package (0.105) installs the agent binary at `<pkg>/libexec/polkit-gnome-authentication-agent-1`, matching the task's assumed path exactly (not `lib/polkit-gnome/...`).
- **Critical grounding for the "how to reference it" research question**: `ls /run/current-system/sw` on the live system shows only `bin`, `sbin`, `lib`, `etc`, `share` are linked into the system profile — **`libexec` is not linked**. This means the task's own suggested phrasing `<polkit_gnome>/libexec/...` cannot be resolved as `/run/current-system/sw/libexec/...` the way `/run/current-system/sw/bin/swaybg` works for other system packages; a bare `environment.systemPackages` install of `pkgs.polkit_gnome` gives **no usable invocation path** for `spawn-at-startup` at all (static `config.kdl` cannot reference a raw, unstable `/nix/store/<hash>-polkit-gnome-0.105/libexec/...` path since it would break on every polkit_gnome update).
- **Recommendation**: wrap the libexec binary using the `writeShellScriptBin` pattern **already established twice in this exact file** (`modules/system/packages.nix:204-219`, for `zathura` and `sioyek`), replacing the commented line at `packages.nix:32`:
  ```nix
  (pkgs.writeShellScriptBin "polkit-gnome-authentication-agent-1" ''
    exec ${pkgs.polkit_gnome}/libexec/polkit-gnome-authentication-agent-1 "$@"
  '')
  ```
  This produces a stable `bin/polkit-gnome-authentication-agent-1` that **is** linked into `/run/current-system/sw/bin/`, so `config/config.kdl` can spawn it the same way it already spawns `swaybg`:
  ```
  spawn-at-startup "/run/current-system/sw/bin/polkit-gnome-authentication-agent-1"
  ```
  This is a system package (matches the task's specified location, `modules/system/packages.nix`), requires no change to how `config.kdl` is wired into home-manager (stays a plain static file), and needs no `nixos-rebuild`-then-`home-manager switch` ordering subtlety beyond the two commands already required for the other fixes.
- **Ruled out**: relying on XDG autostart auto-detection (niri's `niri.service` unit has `Wants=xdg-desktop-autostart.target`, and the niri wiki states GNOME-configured autostart apps "just work" in niri). Inspected the actual shipped `.desktop` file (`<pkg>/etc/xdg/autostart/polkit-gnome-authentication-agent-1.desktop`): it declares `OnlyShowIn=GNOME;XFCE;Unity;`, which niri's `XDG_CURRENT_DESKTOP=niri` does not match, so systemd's xdg-autostart generator will **not** start it under niri — an explicit `spawn-at-startup` (or a hand-written systemd unit) is required, exactly as the task assumes.
- **Also confirmed no double-agent risk under GNOME**: the same `.desktop` file also carries `AutostartCondition=GNOME3 unless-session gnome`, meaning upstream polkit-gnome itself is designed to stay dormant when the real `gnome` session is active (gnome-shell supplies its own agent) — so installing `pkgs.polkit_gnome` system-wide is safe for the GNOME session even though the binary is now present on the system; it simply never launches there via any automatic mechanism, only via the niri-scoped `spawn-at-startup` line.
- **Alternative not recommended for this task**: home-manager ships a full `services.polkit-gnome` module (`modules/services/polkit-gnome.nix` in the flake input) that creates `systemd.user.services.polkit-gnome` with `WantedBy = [ "graphical-session.target" ]` (hardcoded, no `targets` override option unlike waybar). Using it as-is would reintroduce the same shared-target double-start risk documented for waybar above (worse, since it's hardcoded and can't be scoped to `niri.service` without a `mkForce` override of the whole unit). Not recommended without an override; `spawn-at-startup` avoids the issue entirely and matches the task's explicit direction.

### 3. Wallpaper broken (config.kdl:250)

- `config/config.kdl:250`: `spawn-at-startup "/run/current-system/sw/bin/swaybg" "-i" "~/.wallpapers/current" "-m" "fill"`.
- Niri's bundled wiki (`Configuration:-Miscellaneous.md`, `#spawn-at-startup`) states explicitly: *"This option works the same way as the `spawn` key binding action... niri does not use a shell to run commands, which means that you need to manually separate arguments."* The `spawn` action doc (`Configuration:-Key-Bindings.md:209`) reiterates: *"niri does not use a shell to run commands"* — so `~` is passed to `swaybg` literally, never expanded by niri itself, and `swaybg` does not do its own tilde-expansion for the `-i` argument.
- Confirmed via `grep -rn "wallpapers"` that no module creates `~/.wallpapers/current`; the only wallpaper artifact anywhere in the repo is `/etc/wallpapers/riverside.jpg`, installed system-wide via `modules/system/desktop.nix:33` (`environment.etc."wallpapers/riverside.jpg".source = ../../wallpapers/riverside.jpg;`) and already used for the GDM background/lockscreen (`desktop.nix:20-31`).
- **Recommendation**: change line 250 to the absolute path already used elsewhere in the same file:
  ```
  spawn-at-startup "/run/current-system/sw/bin/swaybg" "-i" "/etc/wallpapers/riverside.jpg" "-m" "fill"
  ```
  This is strictly correct and simplest: even a `spawn-sh-at-startup "swaybg -i ~/.wallpapers/current -m fill"` rewrite (available since niri 25.08, present in the installed 26.04) would only fix the tilde-expansion half of the bug — the target path still would not exist, so `swaybg` would still fail. The absolute-path fix addresses both the non-expansion behavior and the missing-file root cause in one change, and keeps the wallpaper source in sync with what GDM already displays (visual consistency across GDM login screen and both sessions).

## Decisions
- Waybar: `spawn-at-startup "waybar"` (bare name) added to `config/config.kdl`'s autostart section — not `programs.waybar.systemd.enable`.
- Polkit: uncomment/replace `modules/system/packages.nix:32` with a `writeShellScriptBin "polkit-gnome-authentication-agent-1"` wrapper around `${pkgs.polkit_gnome}/libexec/polkit-gnome-authentication-agent-1`, then `spawn-at-startup "/run/current-system/sw/bin/polkit-gnome-authentication-agent-1"` in `config.kdl` — not a raw `pkgs.polkit_gnome` install (no usable path) and not the home-manager `services.polkit-gnome` module (hardcoded shared-target risk).
- Wallpaper: change `config.kdl:250`'s image argument from `~/.wallpapers/current` to the absolute `/etc/wallpapers/riverside.jpg` — not a `spawn-sh-at-startup` wrapper (would not fix the missing-file root cause).

## Risks & Mitigations
- **Risk**: forgetting `nixos-rebuild switch` before `home-manager switch` (or vice versa) — the polkit wrapper package lives in `environment.systemPackages` (system rebuild) while `config.kdl` is a home-manager-managed file (home-manager switch). Both must land before the polkit `spawn-at-startup` line will resolve to a valid binary. **Mitigation**: implementation plan should sequence `sudo nixos-rebuild switch --flake .#<host>` before `home-manager switch --flake .#<user>` (or use whatever combined rebuild script this repo already uses, if any — not found in this scope, worth checking `update.sh`/host-level rebuild scripts during planning).
- **Risk**: waybar's `tray` module (`waybar.nix:89-91`) depends on a working StatusNotifierItem/D-Bus tray host; this is unaffected by the startup fix but worth a smoke-check during VERIFY (tray icons appearing, not just the bar itself).
- **Risk (low)**: the stale autostart-section comment (`config.kdl:245-247`) claiming "no manual spawns needed" may confuse future edits if left in place after this fix adds two new manual spawns contradicting it. Recommend a one-line comment correction alongside the fix (in scope for the implementation plan, not a separate task).
