## Executive Summary

- The `gmail-oauth2-refresh.service` fails because OAuth2 tokens have never been initialized
- Error message: "No refresh token found. Please run: himalaya account configure gmail"
- The keyring entries required by the refresh script do not exist in GNOME Keyring
- Himalaya is properly installed with OAuth2 and keyring support (v1.1.0)
- The Home Manager configuration correctly defines the service, timer, and refresh script
- **Solution**: Run `himalaya account configure gmail` to complete OAuth2 setup via the wizard

## Context & Scope

The `gmail-oauth2-refresh.service` is a Home Manager-managed systemd user service that runs every 45 minutes (with 5-minute random delay) to refresh Gmail OAuth2 access tokens. The service is triggered by `gmail-oauth2-refresh.timer` and invokes a custom shell script (`refresh-gmail-oauth2`) defined in `home.nix`.

### Service Architecture

```
gmail-oauth2-refresh.timer (every 45 min)
    |
    v
gmail-oauth2-refresh.service
    |
    v
refresh-gmail-oauth2 script
    |
    +--> Reads tokens from GNOME Keyring (secret-tool)
    |    - service: himalaya-cli
    |    - username: gmail-smtp-oauth2-refresh-token
    |    - username: gmail-smtp-oauth2-access-token
    |    - username: gmail-smtp-oauth2-client-secret
    |
    +--> Calls Google OAuth2 token endpoint
    |
    +--> Stores refreshed access token back to keyring
```

## Findings

### Root Cause: Missing OAuth2 Tokens

The service fails immediately because the refresh token does not exist in the keyring:

```
$ secret-tool lookup service himalaya-cli username gmail-smtp-oauth2-refresh-token
(no output, exit code 1)
```

The refresh script checks for this token first and exits with an error message:

```bash
if ! secret-tool lookup service himalaya-cli username gmail-smtp-oauth2-refresh-token >/dev/null 2>&1; then
  echo "No refresh token found. Please run: himalaya account configure gmail"
  exit 1
fi
```

## Decisions

1. The service implementation is correct; no code changes are needed
2. The issue is initialization, not configuration
3. User must complete OAuth2 wizard to populate keyring entries

## Recommendations

### Primary Solution: Initialize OAuth2 Tokens

Run the himalaya account configuration wizard:

```bash
himalaya account configure gmail
```

This wizard will:
1. Prompt for OAuth2 client secret (if not already stored)
2. Open a browser for Google OAuth2 consent flow
3. Exchange authorization code for access and refresh tokens
4. Store all tokens securely in GNOME Keyring

### Post-Initialization Steps

After running the wizard:

1. **Verify tokens exist**:
   ```bash
   secret-tool lookup service himalaya-cli username gmail-smtp-oauth2-refresh-token
   # Should output the refresh token
   ```

2. **Test the refresh service manually**:
   ```bash
   systemctl --user start gmail-oauth2-refresh.service
   systemctl --user status gmail-oauth2-refresh.service
   ```

3. **Clear failed state**:
   ```bash
   systemctl --user reset-failed gmail-oauth2-refresh.service
   ```

4. **Verify timer is working**:
   ```bash
   systemctl --user list-timers | grep gmail
   ```

### Alternative: Service Resilience Improvement (Optional)

Consider modifying the service to avoid failed status when tokens are not configured:

```nix
# In systemd.user.services.gmail-oauth2-refresh.Service:
Service = {
  Type = "oneshot";
  ExecStart = "${config.home.homeDirectory}/.nix-profile/bin/refresh-gmail-oauth2";
  # Add to prevent failed status:
  SuccessExitStatus = "1";  # Treat exit 1 as success
  # Or use a wrapper that checks and skips gracefully
};
```

However, keeping the failure visible is preferable for awareness that setup is incomplete.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| OAuth2 consent expires/revoked | Low | Tokens stop working | Re-run wizard when needed |
| Refresh token rotation | Medium | Old token invalidated | Script handles token rotation in response |
| GNOME Keyring locked | Low | Service fails | Ensure keyring unlocked at login |
| Client ID/secret mismatch | Low | Auth fails | Verify GMAIL_CLIENT_ID env var matches Google Console |
