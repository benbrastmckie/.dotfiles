## Findings

### Existing Configuration — `.github/workflows/ci.yml`

Current full contents (27 lines):

```yaml
name: nix-flake-check

on:
  push:
  pull_request:

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  flake-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v7
      - uses: cachix/install-nix-action@v31
        with:
          github_access_token: ${{ secrets.GITHUB_TOKEN }}
      - run: nix flake check
      - name: statix (non-blocking)
        run: nix develop --command statix check || true
      - name: deadnix (non-blocking)
        # hosts/*/hardware-configuration.nix is auto-generated (nixos-generate-config) and carries
        # a "Do not modify this file!" header — excluded by path (task 101) since deadnix has no
        # config-file mechanism (unlike statix.toml above the flake check step).
        run: nix develop --command deadnix --exclude 'hosts/*/hardware-configuration.nix' . || true
```

**Shielding mechanism**: Both lint steps use a trailing `|| true` on the shell command itself (lines 21 and 26), not the GitHub Actions `continue-on-error:` step key. This means:
- `nix develop --command statix check || true` — if `statix check` exits non-zero (findings present), the `|| true` makes the overall shell command exit 0, so the step (and job) never sees a failure.
- Same pattern for the deadnix line.

There is no `continue-on-error: true` anywhere in the file — the task description mentions this as one possible shielding mechanism, but the actual mechanism in this repo is the shell-level `|| true`.

## Decisions

- Treat the shielding mechanism as shell-level `|| true` (not `continue-on-error:`), since that is what the file actually contains — the task description's phrasing ("continue-on-error / warn-only shielding") is generic language covering either mechanism, and this repo uses the `|| true` form.
- No changes to `statix.toml` or the deadnix `--exclude` flag are needed or should be made; they already satisfy the "preserve exclusions" requirement from task 101 phase 7.
- Recommend (not strictly required) renaming the two step names from `statix (non-blocking)` / `deadnix (non-blocking)` to `statix` / `deadnix` as part of the same edit, since leaving the stale "(non-blocking)" label on a now-blocking step is misleading to future readers of the workflow file.

## Risks & Mitigations

- **Risk**: A future contributor's PR introduces a new lint finding after this change, and the build starts failing where it previously would not have. **Mitigation**: this is the intended, desired behavior per the task description ("gate the workflow, so future lint regressions fail the build") — not a risk to mitigate, but the explicit goal.
- **Risk**: `nix develop --command` invocations are slow in CI (flake evaluation + shell entry per step) — three separate `nix develop` invocations (flake check implicitly, then statix, then deadnix) each pay shell-entry cost. **Mitigation**: out of scope for task 102 (no performance requirement stated); not a correctness risk, only a minor CI runtime consideration. Not blocking.
- **Risk**: Local `nix develop` verification succeeded but GitHub Actions runners could differ in nixpkgs pin or statix/deadnix version, producing different findings on CI vs. locally. **Mitigation**: the flake lock pins exact versions of `pkgs.statix` and `pkgs.deadnix` used both locally and in CI (same `nix develop --command` invocation, same flake), so behavior should be identical modulo transient network/build issues unrelated to lint content.
