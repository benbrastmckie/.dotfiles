# Phase 3 Handoff: census freshness rename/deletion-aware signal

**Immediate Next Action**: Proceed to Phase 4 — update `wrapper-contracts.md` §13 and
`domain/staleness-detection.md` with the new field/verdict contract.

**Current State**: Phases 1-3 complete. `modules/home/email/agent-tools/census.nix` now emits
`renamed=`/`removed=`/`added=` fields before the trailing `[ok|STALE]` bracket, and the verdict
flips STALE on either the pre-existing count-tolerance check or the new exact-set check
(`SETDIFF_TOL=0`). Plan file phase markers are current through Phase 3.

**Key Decisions Made**:
- `SETDIFF_TOL=0` — live back-to-back runs showed perfectly stable (non-jittery) nonzero counts,
  so no slack was warranted.
- Basename reduction for the indexed side is done inline in the awk join (`sub(/.*\//, "", base)`)
  rather than a separate `sed` pipe stage — functionally identical, fewer processes.
- Live verification used the built derivation's store path directly rather than a full
  `home-manager switch`, since switching the live system generation is out of scope for a
  config-file change task.

**Deviations from Plan**: See inline `*(deviation: ...)*` annotations on Phase 1 and Phase 3
checklist items in the plan file — all are "altered" (equivalent implementation, or verification
method substitution), none are skips.

**What NOT to Try**: Do not switch the query form to `notmuch count`/`tag:`-based — reintroduces
the `search.exclude_tags` false-positive trap documented at wrapper-contracts.md §13. Do not run
`email-reindex` as part of this task (explicit Non-Goal).
